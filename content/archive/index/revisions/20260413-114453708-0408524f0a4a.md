{% sectionCopy eyebrow="what it is" title="a site system with one obvious path from draft to production." %}
PageQuarry is for people who want the website to stay understandable after the launch rush is over. It gives you clear seams for brand, layout, and content without turning every routine edit into a tiny software consultancy.

You fork the repo, set the site identity, write in markdown, validate changes, and ship static output. The point is not infinite flexibility. The point is staying coherent.
{% /sectionCopy %}

{% process eyebrow="how it works" title="configure it, write in markdown, publish with guardrails." %}
{% step title="set the site identity" body="Put the name, metadata, nav, footer, and primary action in one obvious config surface." /%}
{% step title="edit through the content pipeline" body="Draft and revise content in markdown, then validate and accept it through the CLI instead of freehanding production files." /%}
{% step title="ship from generated state" body="The runtime rebuilds from accepted output, so the published site stays deterministic and easier to trust." /%}
{% /process %}

{% sectionCopy eyebrow="why it feels different" title="it is opinionated where most CMSes get mushy." tone="subtle" %}
Most CMSes promise freedom and quietly hand you entropy. PageQuarry does the opposite. It narrows the editing surface so humans and agents can keep making changes without slowly turning the site into soup.

That means visible history, shared blocks, recoverable mistakes, and fewer hidden places for complexity to rot.
{% /sectionCopy %}
